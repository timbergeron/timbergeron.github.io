NetQuake vs QuakeWorld

The old rule of thumb still holds: classic NetQuake is more server-bound, friction-heavy, and awkward around walls, while classic QuakeWorld is built to replay movement cleanly and preserve speed through hops.

Original summary compiled by Tonik. This version expands it after checking the current QSS-M tree on the NetQuake side and ezQuake on the QuakeWorld side.

The Big Split

NetQuake / QSS-M default path

QSS-M still carries the classic NetQuake server movement path in sv_phys.c: SV_Physics_Client drives SV_WalkMove, and wall drag is applied through SV_WallFriction. Its default sv_nqplayerphysics 1 setting keeps that authoritative NQ feel.

QuakeWorld / ezQuake path

ezQuake is built around shared pmove. The server fills movevars in sv_user.c, runs PM_PlayerMove(), and the client reruns the same usercmd stream in cl_pred.c and cl_main.c so movement can be predicted locally.

Why that changes the feel

NQ tends to expose server timing, pre-jump friction, edge drag, and wall stickiness. QW turns movement into a repeatable command replay problem, which is why it supports true bunnyhopping, tighter tele exits, and much more reliable strafe timing under latency.

The Bunnyhop Difference in One Glance

NQ-leaning order

PM_Friction();
PM_CheckJump();

QSS-M exposes both orders, but its default pm_bunnyfriction 1 keeps the NetQuake style: speed is taxed before the jump, so chained hops do not keep building forever.

QW order

PM_CheckJump();
PM_Friction();

ezQuake's PM_PlayerMove() does the QuakeWorld order directly. That missing friction frame is the core reason QW bunnyhops keep momentum and reward exact strafing.

Source-Backed Differences

Topic NetQuake / QSS-M QuakeWorld / ezQuake Practical result
Movement authority Classic movement still lives in the server-side walk physics path, and QSS-M only switches to shared pmove when prediction is explicitly enabled. Shared pmove is the normal design: server and client both run the same movement code. NQ feels more tied to server cadence; QW feels immediate because the client can rehearse the move before the server reply lands.
Prediction Optional. QSS-M even prints a message when server movement prediction becomes active, which underlines that it is a mode rather than the baseline. Core behavior. ezQuake uses CL_PredictMove() and CL_PredictUsercmd(); prediction is only absent if you deliberately disable it. QW movement is much easier to place precisely in online play.
Jump and friction order Defaults to the NQ friction-first hop through pm_bunnyfriction. Checks the jump before applying friction. This is the heart of the bunnyhop split: NQ burns speed on takeoff, QW keeps it.
Edge friction pm_edgefriction is exposed, and QSS-M can choose between an NQ-style line test and the QW-style edge-box behavior. PM_Friction() doubles ground friction when the forward trace hangs over a dropoff. Both punish sloppy ledge movement, but the trigger geometry differs, so the slowdown starts at different moments.
Wall friction Classic server physics still calls SV_WallFriction after step traces. No equivalent heavy wall-friction pass exists in ezQuake's pmove.c. NQ wall scrapes feel sticky; QW lines along walls keep more speed.
Step and ramp handling Classic step logic remains, but QSS-M adds many compatibility knobs such as pm_slidefix, pm_stepdown, pm_slidyslopes, pm_airstep, and pm_walljump. Modern QW exposes competitive movement switches such as pm_slidefix, pm_airstep, pm_pground, and pm_rampjump. Modern ports can blur the old line, but QW still starts from the more air-control-friendly baseline.
Timing model Shared pmove uses cmd.seconds * gamespeed, while the classic NQ path is still closely tied to server physics frames. PM_PlayerMove() runs on cmd.msec * 0.001, and the same usercmds are replayed locally. QW is engineered around exact usercmd replay; NQ historically leaks more tick-rate quirks into movement.

Classic Gameplay Differences Players Usually Mean

These are the long-standing ruleset differences people usually mean when they say "NQ physics" versus "QW physics." Modern engines can emulate parts of both.

Topic NetQuake QuakeWorld What it changes
Bunnyhopping There is a friction frame before the hop, so speed gain is limited and depends much more on server timing. The friction frame is effectively absent, so acceleration can be chained continuously. QW rewards repeatable speed-building routes; NQ rewards cleaner, more conservative lines.
Teleport exits The classic backpedal lock after a teleport forces a committed turn or reposition. No comparable lock, so you can re-enter or backpedal much more aggressively. QW tele routes are faster and less committal.
Projectile launch ("newmis") Projectiles launch from the old origin logic. Projectiles are nudged forward by the player's velocity on spawn. That changes rocket-jump timing, corner shots, and the order in which jump and attack can be buffered.
Edge play Edge slowdown arrives earlier and is usually more noticeable. Edge handling is milder in feel and combines with freer air control. Tight ledges and lip-strafes are less punishing in QW.
Ramps and slopes Vertical speed is easier to preserve, and classic NQ slope behavior can create odd boosts or sticky wall-slope interactions. Ground handling is more normalized, and modern QW servers often expose explicit rampjump toggles. The same ramp can be a speed bump, a boost, or a dead line depending on ruleset.

What Those Rules Change in Real Matches

This is the layer strong players usually care about most: not just what the code says, but what actually changes in live combat, item fights, and team comms.

Rocket confirmation A clean direct rocket is always 110, so "he ate a direct" is reliable information for the second peeker. A clean direct rocket is 100-120, so the same-looking hit does not always leave the same cleanup. KTX rewards tighter synchronized re-peeks and cleaner focus-fire calls. CRMod7 teams usually leave a slightly wider safety margin before collapsing.
Lightning fights LG is a fixed 30-damage single center trace, and QW prediction makes repeated shaft entries more consistent online. Rune burst versus sustain Rune power is heavily server-driven: strength, resistance, haste, and regeneration all scale off cvars. Rune behavior is more binary and default-strong: x2 strength, /2 resistance, fixed haste refires, fixed 0.5s regen ticks. KTX CTF can swing from mild to extreme depending on config. Modern Engines Blur the Old Binary

QSS-M is still NQ-first Its defaults keep classic NetQuake server physics, but it also ships a large compatibility layer: shared prediction, QW-style bunnyhop options, edge-friction switches, walljump, airstep, and more.
ezQuake is still QW-first Its normal path assumes predicted QuakeWorld movement, but it also has NQ compatibility pieces such as cl_nqdemo.c for NetQuake demo playback and server-controlled movement cvars.
Bottom line If you want a one-sentence memory aid: NQ is authoritative, sticky, and friction-first; QW is replayable, speed-preserving, and prediction-first.
